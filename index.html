Asynchronous JS

Async/await

Why do we need asynchronous functions?

Because JavaScript is single-threaded

When we make a long I/O call (like fetch), are we going to block the thread until it is done?
An HTTP request can take any amount of time. The UI could be frozen for a full 2 seconds or maybe even 20!

Asynchronous JS

Async/await

Problem: JavaScript is single-threaded.
Solution: Don't block, and keep code running while the I/O call is processing.
We can register some code to run once the I/O call has resolved.

Component frameworks

Encapsulate, reuse & share

Over the years, several issues became apparent with "normal" frontend development:

  • Imperative DOM manipulation in JavaScript is a pain.
    > You can view the DOM as a global variable that can be mutated from anywhere in the code. Yikes.
  • App is split in HTML/CSS/JS but this isn't how humans think about a UI
  • Lots of boilerplate and security pitfalls (XSS)
  • Hard to avoid duplicate code

Component frameworks have therefore become very popular.

Component frameworks

Encapsulate, reuse & share

Component frameworks have therefore become very popular.

  • Angular (2010) by Google
  • React (2013) by Facebook
  • Vue (2014) by Evan You.
    Light-weight, minimal Angular alternative
  • Svelte (2018) by Rich Harris.
    Because all the above are "lacking/non-performant"

Each has its own, non-standard syntax to enhance HTML/CSS/JS.

Adding a buildstep

Using frameworks & tools

Component frameworks are often available as just JS libraries

However, nowadays, it is common to have compilers/transpilers/bundlers that allow for easier syntax/complex features

This means you write your website in an easier way than raw HTML/CSS/JS, and then compile it to HTML/CSS/JS. This is a buildstep.

You mostly use NodeJS for this, which runs JS outside the browser
